## 1. Purpose and Scope

This constitution governs the implementation of Phase 5 of the Todo AI Chatbot project, which introduces:
- **Advanced Features**: Recurring tasks, reminders, priorities, tags, search/filter/sort
- **Event-Driven Architecture**: Kafka-based messaging with Dapr abstraction
- **Cloud Deployment**: Production-ready deployment on Azure AKS or Google GKE
- **Full Observability**: Monitoring, logging, tracing, and alerting
- **CI/CD Automation**: GitHub Actions for continuous deployment

This document is the **supreme authority** for all implementation decisions. Every agent, script, configuration, and code generation must comply with this constitution.

### 2.3 Event-Driven Architecture Requirements
1. **Kafka Topics**: All async operations must use these topics:

   - `reminders`: Reminder notifications and scheduling
   - `task-updates`: Real-time task updates for frontend

   - **Pub/Sub**: Kafka messaging (pubsub.kafka component)

   - **Secrets**: API keys and credentials (secretstores.kubernetes component)
   - **Service Invocation**: Frontend-to-backend communication (service-to-service calls)

3. **No Direct Dependencies**: Code must NOT directly import Kafka, PostgreSQL, or cloud SDKs. All interactions via Dapr HTTP/gRPC APIs.

## 3. Key Principles

### Principle 1: Event-Driven Microservices with Kafka + Dapr
- **Decoupling**: Services communicate via Kafka topics, not direct HTTP calls
- **Async Operations**: Reminders, recurring tasks, notifications are event-driven
- **Example**: When a task is created, backend publishes to `task-events` topic; reminder-worker subscribes and schedules reminders via Dapr Jobs API

### Principle 2: Full Dapr Sidecar for All Infrastructure
- **Abstraction**: Dapr sidecars handle all infrastructure concerns (messaging, state, secrets, jobs)
- **Portability**: Swap Kafka for RabbitMQ or PostgreSQL for MongoDB by changing Dapr component YAML only
- **Example**: Backend calls `http://localhost:3500/v1.0/publish/pubsub-kafka/task-events` instead of Kafka SDK

### Principle 3: Cloud-Native Resilience and Observability
- **Resilience**: Liveness/readiness probes, auto-scaling (HPA), self-healing (restart policies)
- **Observability**: Metrics (Prometheus), logs (Loki/cloud), traces (Dapr tracing), events (Kubernetes events)
- **Example**: Backend has `/health` and `/ready` endpoints; HPA scales on CPU >70%

### Principle 4: Free-Tier Optimization
- **Cost Awareness**: Use only free credits and trials; no paid services beyond trials
- **Resource Limits**: Set CPU/memory requests and limits to stay within free tier quotas
- **Example**: Use Azure $200 credits for AKS, Redpanda Cloud free tier for Kafka, Neon DB free tier for PostgreSQL

### Principle 5: Documentation-First Approach
- **Sequence**: constitution.md â†’ phase5-spec.md â†’ plan.md â†’ tasks.md â†’ code
- **Traceability**: Every code file must reference the task ID and spec section it implements
- **Example**: `backend/src/events/task-publisher.ts` has header comment: `// Task: T5.3.2 - Implements phase5-spec.md Section 4.2 (Kafka Publisher)`

### Principle 6: Testing and Validation
- **Test Coverage**: Every feature must have unit tests, integration tests, and E2E tests
- **CI/CD Gates**: All tests must pass before deployment to any environment
- **Example**: GitHub Actions workflow runs `npm test`, `npm run test:integration`, `npm run test:e2e` before deploying

## 4. Architecture Requirements

### 4.1 System Components

| Component | Technology | Purpose | Dapr Integration |
|-----------|-----------|---------|------------------|
| Backend API | Node.js/Express or Python/FastAPI | REST API, business logic | Dapr sidecar for Pub/Sub, State, Jobs |
| Frontend | React/Next.js | User interface | Dapr sidecar for Service Invocation |
| Reminder Worker | Node.js or Python | Processes reminder events | Dapr sidecar for Pub/Sub, Jobs |
| Database | PostgreSQL (Neon DB) | Persistent storage | Dapr State Store component |
| Message Broker | Kafka (Redpanda/Strimzi) | Event streaming | Dapr Pub/Sub component |
| State Store | PostgreSQL | Dapr state management | Dapr State Store component |
| Secrets | Kubernetes Secrets | Credentials, API keys | Dapr Secret Store component |

### 4.2 Event Flow

```
User Action (Frontend)
  â†“
Backend API (Dapr Service Invocation)
  â†“
Publish Event to Kafka (Dapr Pub/Sub)
  â†“
Kafka Topic (task-events, reminders, task-updates)
  â†“
Subscriber (Reminder Worker via Dapr Pub/Sub)
  â†“
Schedule Job (Dapr Jobs API)
  â†“
Execute Reminder (Publish to reminders topic)
  â†“
Frontend Receives Update (WebSocket or polling)
```

### 4.3 Dapr Components

All Dapr components must be defined in `dapr-components/` directory:

1. **pubsub-kafka.yaml**: Kafka Pub/Sub component
   - Brokers: Redpanda Cloud or Strimzi cluster
   - Topics: task-events, reminders, task-updates
   - Consumer groups: backend-group, worker-group

2. **statestore-postgresql.yaml**: PostgreSQL State Store
   - Connection string: From Kubernetes Secret
   - Table: dapr_state
   - Metadata table: dapr_state_metadata

3. **jobs-scheduler.yaml**: Jobs API component
   - Backend: PostgreSQL state store
   - Max concurrent jobs: 10
   - Retry policy: Exponential backoff

4. **secretstore-kubernetes.yaml**: Kubernetes Secret Store
   - Namespace: todo-app (or default)
   - Secrets: db-credentials, kafka-credentials, api-keys

### 4.4 Advanced Features Implementation

| Feature | Implementation | Kafka Topic | Dapr Component |
|---------|---------------|-------------|----------------|
| Recurring Tasks | Cron expression parser + Dapr Jobs API | task-events | jobs-scheduler |
| Reminders | Scheduled jobs via Dapr Jobs API | reminders | jobs-scheduler, pubsub-kafka |
| Priorities | Enum field (low/medium/high/urgent) + UI dropdown | task-events | state.postgresql |
| Tags | Many-to-many relationship + tag chips UI | task-events | state.postgresql |
| Search | Full-text search on title/description | task-updates | state.postgresql |
| Filter | Query params (status, priority, tags) | task-updates | state.postgresql |
| Sort | Query params (createdAt, priority, dueDate) | task-updates | state.postgresql |

## 5. Deployment Requirements

### 5.1 Local Deployment (Minikube)

**Prerequisites:**
- Minikube 1.32+
- kubectl 1.28+
- Helm 3.12+
- Dapr CLI 1.12+
- Docker Desktop

**Steps:**
1. Start Minikube: `minikube start --cpus=4 --memory=8192 --driver=docker`
2. Install Dapr: `dapr init -k`
3. Deploy Kafka (Redpanda or Strimzi): `helm install redpanda redpanda/redpanda`
4. Deploy PostgreSQL: `helm install postgres bitnami/postgresql`
5. Apply Dapr components: `kubectl apply -f dapr-components/`
6. Deploy backend: `helm install backend charts/backend/`
7. Deploy frontend: `helm install frontend charts/frontend/`
8. Deploy reminder-worker: `helm install worker charts/reminder-worker/`
9. Port-forward: `kubectl port-forward svc/frontend 3000:3000`
10. Test: `curl http://localhost:3000/health`

### 5.2 Cloud Deployment (Azure AKS)

**Prerequisites:**
- Azure account with $200 free credits
- Azure CLI 2.55+
- kubectl 1.28+
- Helm 3.12+

**Steps:**
1. Create AKS cluster: `az aks create --name todo-cluster --resource-group todo-rg --node-count 3 --enable-managed-identity`
2. Get credentials: `az aks get-credentials --name todo-cluster --resource-group todo-rg`
3. Install Dapr: `dapr init -k`
4. Deploy Kafka (Redpanda Cloud or Strimzi): Use Redpanda Cloud free tier or `helm install strimzi strimzi/strimzi-kafka-operator`
5. Deploy PostgreSQL: Use Azure Database for PostgreSQL (free tier) or Neon DB
6. Apply Dapr components: `kubectl apply -f dapr-components/`
7. Deploy with Helm: `helm install backend charts/backend/ --set image.tag=latest`
8. Setup ingress: `kubectl apply -f k8s/ingress.yaml` (with cert-manager for HTTPS)
9. Configure DNS: Point domain to ingress IP
10. Verify: `curl https://todo.yourdomain.com/health`

### 5.3 Cloud Deployment (Google GKE)

**Prerequisites:**
- Google Cloud account with $300 free credits
- gcloud CLI 460+
- kubectl 1.28+
- Helm 3.12+

**Steps:**
1. Create GKE cluster: `gcloud container clusters create todo-cluster --num-nodes=3 --zone=us-central1-a`
2. Get credentials: `gcloud container clusters get-credentials todo-cluster --zone=us-central1-a`
3. Install Dapr: `dapr init -k`
4. Deploy Kafka: Use Confluent Cloud free tier or Strimzi
5. Deploy PostgreSQL: Use Cloud SQL (free tier) or Neon DB
6. Apply Dapr components: `kubectl apply -f dapr-components/`
7. Deploy with Helm: `helm install backend charts/backend/`
8. Setup ingress: Use GKE Ingress with Google-managed SSL certificates
9. Configure DNS: Point domain to ingress IP
10. Verify: `curl https://todo.yourdomain.com/health`

## 6. CI/CD Requirements

**File:** `.github/workflows/deploy.yml`

**Triggers:**
- Push to `main` branch
- Pull request to `main` branch

**Jobs:**
1. **Build**:
   - Checkout code

   - Build Docker images (backend, frontend, worker)

   - Run unit tests (`npm test`)
   - Run integration tests (`npm run test:integration`)
   - Run E2E tests (`npm run test:e2e`)
   - Generate code coverage report
   - Upload coverage to Codecov

3. **Deploy to Staging**:
   - Deploy to staging cluster (AKS/GKE)
   - Run smoke tests
   - Validate health endpoints

4. **Deploy to Production** (manual approval):
   - Deploy to production cluster
   - Run smoke tests

- **Versioning**: Use semantic versioning (v1.0.0, v1.1.0, etc.)

- **Notifications**: Slack/Discord notifications on deployment success/failure

## 7. Monitoring and Observability

### 7.1 Metrics (Prometheus + Grafana)

**Metrics to Track:**
- Request rate (req/sec)
- Response time (p50, p95, p99)
- Error rate (4xx, 5xx)
- CPU usage (%)
- Memory usage (MB)
- Kafka consumer lag
- Dapr sidecar health

**Dashboards:**
- Application dashboard (request rate, latency, errors)
- Infrastructure dashboard (CPU, memory, disk, network)
- Kafka dashboard (throughput, lag, partition distribution)
- Dapr dashboard (sidecar health, component status)

### 7.2 Logging (Loki or Cloud Logging)

**Log Levels:**
- ERROR: Critical errors requiring immediate attention
- WARN: Warnings that may indicate issues
- INFO: Informational messages (e.g., "Task created")
- DEBUG: Detailed debugging information

**Log Aggregation:**
- Local: Loki + Promtail
- Cloud: Azure Log Analytics or GKE Logging

### 7.3 Tracing (Dapr Tracing)

**Tracing Backend:**
- Local: Zipkin
- Cloud: Azure Application Insights or GKE Cloud Trace

**Trace Spans:**
- HTTP requests (frontend â†’ backend)
- Kafka publish/subscribe events
- Dapr state store operations
- Dapr service invocations

### 7.4 Alerting

**Alert Rules:**
- Error rate > 5% for 5 minutes
- Response time p95 > 1000ms for 5 minutes
- CPU usage > 80% for 10 minutes
- Memory usage > 90% for 5 minutes
- Kafka consumer lag > 1000 messages

**Alert Channels:**
- Email
- Slack/Discord
- PagerDuty (optional)
